# LSEG DSM AI Assistant

An internal AI assistant for LSEG DSM operational workflows. Answers questions over SOP policy documents and incident tickets using Retrieval-Augmented Generation (RAG) via Amazon Bedrock, with deterministic handling of structured queries (counting and filtering).

---

## Prerequisites

| Requirement | Version |
|---|---|
| Python | 3.12+ |
| [uv](https://docs.astral.sh/uv/getting-started/installation/) | latest |
| AWS account | with Bedrock access enabled |
| AWS CLI | optional (for credential setup) |

## Setup

### 1. Clone and install dependencies

```bash
git clone <repo-url>
cd lseg-ai-assistant
uv sync
```

### 2. Enable Bedrock model access

In the **AWS Console**, navigate to:
**Amazon Bedrock → Model access → Modify model access**

> Make sure you are in the correct AWS region (top-right corner)

Enable both models:
- `Amazon → Titan Text Embeddings V2` (`amazon.titan-embed-text-v2:0`)
- `Anthropic → Claude 3.7 Sonnet` (`anthropic.claude-3-7-sonnet-20250219-v1:0`)

Click **Save** — approval is usually instant.

### 3. Configure AWS credentials

**Option A — AWS CLI (recommended)**
```bash
brew install awscli        # macOS
aws configure
# AWS Access Key ID:     <your-key>
# AWS Secret Access Key: <your-secret>
# Default region name:   eu-west-2
# Default output format: json
```

**Option B — Environment variables**
```bash
export AWS_ACCESS_KEY_ID=your_access_key_id
export AWS_SECRET_ACCESS_KEY=your_secret_access_key
export AWS_DEFAULT_REGION=eu-west-2
```

### 4. Configure the region and model

Edit `src/config.py` if needed:
```python
BEDROCK_REGION = os.environ.get("BEDROCK_REGION", "eu-west-2")
LLM_MODEL_ID   = "anthropic.claude-3-7-sonnet-20250219-v1:0"
```

## Running

### Start the API server

```bash
uv run python main.py
```

On **first run** the app builds the FAISS vector index by embedding all 213 document chunks via Bedrock Titan Embeddings — this takes **30–90 seconds**. You'll see:

```
[INFO] loading documents from docs
[INFO] chunking complete: 213 chunks
[INFO] embedding chunks via Bedrock Titan Embeddings v2
[INFO] faiss index built: 213 vectors (dim=1024)
[INFO] Application startup complete.
```

On **subsequent runs** the index loads from `.cache/` instantly.

The server starts on `http://localhost:8000`.
Interactive API docs (Swagger UI): `http://localhost:8000/docs`

## API Reference

### `GET /health`
Returns service status and index size.

```json
{
  "status": "ok",
  "index_loaded": true,
  "index_size": 213
}
```

### `POST /ask`
Submit an operational question. Returns the answer, source citations, confidence score, and anomaly flag.

**Request**
```json
{ "query": "What should happen if a deployment fails in production?" }
```

**Response**
```json
{
  "answer": "According to policy: If a deployment fails, trigger an automatic rollback within 2 minutes... [SOURCE: SOP-002 § Deployment Failure Handling]",
  "source_type": "rag",
  "citations": ["SOP-002 § Deployment Failure Handling", "SOP-002 § Rollback Verification"],
  "confidence": 0.847,
  "anomaly_flagged": false
}
```

**`source_type` values:**

| Value | Meaning |
|---|---|
| `rag` | Answer generated by Claude from retrieved SOP/ticket context |
| `structured-data` | Exact count/filter result from pandas — no LLM involved |
| `guardrail-rejection` | Refused — retrieval confidence too low to answer safely |

## Architecture

```
docs/ (SOPs + tickets)
    │
    ▼
Ingestion (document_loader + chunker)
    │   Section-level SOP chunks + per-row ticket chunks
    ▼
Bedrock Titan Embeddings v2  (amazon.titan-embed-text-v2:0)
    │   1024-dim normalised vectors
    ▼
FAISS IndexFlatIP  (inner product = cosine similarity)
    │
    ▼  on query:
Query Router
├── Structured (count / filter) → pandas DataFrame  [deterministic]
└── Unstructured (Q&A)
        ├── Embed query → Bedrock Titan
        ├── k-NN search → FAISS (top-5)
        ├── Guardrails
        │   ├── Confidence threshold  (reject if score < 0.30)
        │   ├── Domain validation     (warn on cross-domain retrieval)
        │   └── Policy anomaly flag   (MFA social-engineering detection)
        ├── Prompt builder (system + user separation)
        └── Bedrock Claude 3.7 Sonnet → answer with [SOURCE: ...] citations
```

## Design Decisions

### 1. Chunking strategy: Section-level vs sentence-level

**Chosen: Section-level** — SOP documents are split on `##` markdown headers, one chunk per section.

SOP sections are self-contained operational procedures whose numbered steps must stay together. For example, the MFA Reset Policy section contains both the prohibition and the exception conditions; splitting at the sentence level would force the retriever to find both halves independently, risking incomplete or misleading answers.

**Trade-off:** Larger chunks mean more tokens sent to the LLM per call (higher cost, higher latency). For a low-volume internal tool this is acceptable. A sentence-level approach with cross-chunk context stitching would be needed at scale.

### 2. Vector store: FAISS (local) vs Amazon OpenSearch Serverless

**Chosen: FAISS locally** with OpenSearch Serverless documented in `infra/opensearch_setup.py`.

The assignment requires demonstrating embedding creation, retrieval logic, and LLM invocation explicitly. FAISS keeps these components visible and runnable without additional infrastructure. OpenSearch Serverless is the production-grade replacement — the retriever's `search()` interface is identical in both cases, so the swap requires changing only one module.

**Trade-off:** FAISS is an in-process index (no persistence across replicas, no real-time updates). OpenSearch Serverless solves both — documents uploaded to S3 are automatically re-indexed via the Lambda ingestion pipeline without any app restart.

### 3. Structured query handling: Deterministic pandas vs LLM text-to-SQL

**Chosen: Deterministic pandas** — structured queries (count / filter) never touch the LLM.

Questions like "how many Access tickets are Open?" have exactly one correct answer. LLM generation introduces non-determinism, hallucination risk, and unnecessary latency for queries that are trivially computable from the CSV. The result is labelled `source_type: structured-data` so the caller knows it is exact.

**Trade-off:** The regex/keyword parser is brittle to novel phrasing. A production system would use an NL-to-SQL model (e.g. Bedrock Claude with a structured-output schema) for robustness, at the cost of latency and the small risk of an incorrect filter.

### 4. Latency vs accuracy: top-K retrieval depth

**Chosen: k=5** — retrieve the top 5 chunks per query.

Increasing k improves recall (more relevant context reaches the LLM) but raises cost and latency: each extra chunk adds tokens to the prompt and increases the LLM's time-to-first-token. Decreasing k reduces latency but risks missing a relevant section when the query spans multiple SOP areas.

k=5 is set as a configurable environment variable (`TOP_K`) so it can be tuned in production without a code deploy. Latency is tracked per-component in structured logs (`retrieval_ms`, `llm_ms`, `total_ms`) — in practice, Bedrock LLM invocation dominates at ~1.5–3s while FAISS retrieval is <50ms, so k has minimal impact on total latency at this scale.

**Trade-off:** At higher document volumes (tens of thousands of chunks), a larger k combined with a long-context model would improve accuracy at the cost of significantly higher per-query spend. The right balance depends on query volume and acceptable p95 latency SLA.

## Failure Modes and Mitigations

### 1. Retrieval miss → unnecessary refusal
**Scenario:** The user's phrasing differs from the document vocabulary (e.g. "trading system down" instead of "deployment failed") — cosine similarity falls below the threshold and the assistant refuses to answer.

**Mitigation:** Log all `guardrail-rejection` responses to CloudWatch. Use the logged queries to lower the threshold or add domain-specific synonym expansion (rephrasing the query before embedding). The `CONFIDENCE_THRESHOLD` is configurable via environment variable without a code deploy.

### 2. LLM hallucination on edge cases
**Scenario:** Claude ignores the "context only" rule and fabricates a plausible-sounding SOP step that doesn't exist.

**Mitigation:** `temperature=0` makes the output deterministic. The confidence gate prevents the LLM from being called at all when retrieval is weak. In production, Bedrock Guardrails (managed service) can be layered on top as a second content filter.

### 3. Structured query parser miss
**Scenario:** A user asks "tally up the open access ones" — the pattern doesn't match any of the regex triggers, so the query falls through to the RAG pipeline and returns an approximate (non-exact) answer.

**Mitigation:** The fallback to RAG is safe and labelled `source_type: rag` with `confidence` included, so the caller knows it is approximate. Missed patterns are logged for parser improvement without a code deploy.

## AWS Production Architecture

```
S3 (lseg-ai-docs)
    └── S3 PUT trigger
            └── Lambda (lambda_ingestion.py)
                    ├── Bedrock Titan Embeddings v2  (embed chunks)
                    └── OpenSearch Serverless         (upsert vectors)

ALB → ECS Fargate (FastAPI container, this codebase)
            ├── Bedrock Titan Embeddings v2  (embed queries)
            ├── OpenSearch Serverless         (k-NN search)
            └── Bedrock Claude 3.7 Sonnet    (generate answers)

SSM Parameter Store  — OPENSEARCH_ENDPOINT, CONFIDENCE_THRESHOLD, model IDs
CloudWatch Logs      — structured JSON from ECS + Lambda
CloudWatch Metrics   — p95 latency, guardrail hit rate, anomaly flag count
IAM                  — LsegEcsTaskRole, LsegLambdaIngestionRole (see infra/iam_policies.py)
ECR                  — Docker image for ECS Fargate task
```

See `infra/` for boto3 pseudocode covering S3 setup, OpenSearch collection creation, Lambda deployment, and IAM role definitions.
